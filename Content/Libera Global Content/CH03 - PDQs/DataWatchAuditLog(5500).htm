Data Watch Audit Log



Data Watch Audit Log PDQ.

 

Overview

The Data Watch Audit Log helps you see activity that has occurred in the system. This has many uses. For example, it may be used for tracking user activity or restoring data. The Data Watch Audit Log opens in a new window.

 

PDQ Parameters



Parameters for the Data Watch Audit Log PDQ.

 

  • Start Date and End Date - These are the only required parameters, all others are optional. The PDQ will return records for all actions taken within the system within this range. Because the Data Watch Audit Log is constantly collecting new records, these inputs also collect the time to the second.
  • User - This lists all employees with logins for the application. If one is selected, the results will only include actions taken by the selected User. Only one User can be selected at a time.
  • SSN - This is the SSN of the (if applicable) affected by the action. If it is set, the results will only include actions affecting that . Only one SSN can be accepted at a time.
  • Last X Records - By default, this PDQ displays 200 records at a time from most to least recent. To change this, enter the number of records you want the PDQ to load here. This number is capped at 10,000 in order to prevent using excess bandwidth.
  • Include System Events - By default, the results will NOT include system events such as launching forms with the LForm.GetLForm action. To include such events, set this parameter to 'Y'

 

 PDQ Results
  • Date/Time - By default, the results are ordered from most recent (that is, closest to the given End Date) to least recent.
  • Action - The action taken by System 7 at the given Date/Time. Possible values include LForm.SaveField (when changes to a form are saved) and Audit.Write.
  • Table - This is the data table affected by the action.
  • Column - This is the specific column in the Table mentioned above affected by the action.
  • Old Value - If the action changed an existing value in the Table, then the old value will be recorded here. This column is useful if you need to undo a change or restore a deleted case note.
  • Value - This is the new value in the Table and Column that was caused by the action. Examples: A 's new status, a name change or any value entered into a blank form.
  • Primary Key and Primary Key # - A primary key can be a number or a string of text. Whichever type of primary key is used for this action, it will be recorded in the appropriate column.Sometimes the Primary Key may not be the same as the primary key of the data table. The Primary Key in the PDQ may be other useful information, such as the user name used to logon.
  • # and Client Name - This is the SSN and name of the (if applicable) affected by the action. For example, the action could be an update to a form in their ECF.
  • User # and User Name - This is the ID Number and name of the user who performed the action as it is appears on their Employee Information form, not their login. Filtering with this column is useful to confirm whether or not a given user edited a field.
  • Component Name, Component Type and Component # - These identify the component that the action was performed through. If, for example, the action is a change to a 's address on the form, these fields will identify the form.
  • Transaction - Each time a user sends information, whether by updating a field or asking to view a form, they are making a transaction. This number can be used to identify a transaction. Several actions may be the result of the same transaction if they occurred during the same server event. For example, changing the ZIP Code on a form may cause the City and State fields to update as well with a single transaction.
  • Profile - This is the name of the database containing the table that was changed as a result of the action. This may not be recorded for events with the SEC (security event) flag.

 

 Making the Most of the PDQ Results

The information returned can be further filtered just like any PDQ. Use appropriate filtering as your situation requires.

For example:

You may be looking for a deleted case note so it may be restored. To find the case note among the hundreds of actions, try narrowing the Start and End Date/Time range as much as possible.

Next, filter for the who should have the case note on file, the Component Name or Component Number of the form on which the case note is located and the User who accidentally deleted the note.

If you also know the Tables and Flags values that the action of deleting a case note would have, then you could filter using those as well.
