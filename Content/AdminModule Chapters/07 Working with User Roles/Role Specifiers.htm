Role Specifiers

Combination Role Specifier

The Combination Role Specifier allows you to require users to have two User Roles. The user must have both roles to pass this check. This comes in handy if you have multipier tiers or top level groupings (think divisions or departments) of users.

Explicit Role Specifier

The Explicit Role Specifier is just a number specifying the exact role identifier that is allowed. The user must have the specified role to pass this check.

 



Example of an Explicit Role Specifier.

 

For example, if a PDQ has only one Role, then only users with that role will have access to the PDQ. Adding only one role to a Role or ACL field automatically excludes users with all other roles.

Exclusion Role Specifier

The Exclusion Role Specifier is an exclamation point followed by a number specifying the exact role identifier that is NOT allowed. If the user does not have the specified role, they will pass the check. Some caution should be used with this specifier. It can get confusing to think about because the role specifier list is checked as “OR” logic. The exclusion specifier is an immediately failing check (i.e.: if an exclusion check fails – the next specifier is NOT checked and the ACL will return false meaning the user won't have access). For example:

  • If the user roles are “1,3,5,7”, an ACL of “!5” will prevent access because the user is has role 5.
  • If the user roles are “1,3,5,7”, an ACL of “!5,1,3” will still prevent access because the user has role 5.

 


Example of an Exclusion Role Specifier.

 

Note: When you see !0 in an ACL list it means that no users have access to the component or functionality. All users inherently have User Role 0. As a result when you exclude User Role 0 you exclude all users.

Minimum Role Specifier

The Minimum Role Specifier is just a number followed by a plus sign (+) specifying the minimum role identifier that is allowed. The user must have a role greater than or equal to the numeric role to pass the check. This specifier only makes sense when role identifiers are set up in monotonically increasing numbers by security level (i.e., 0 = no access; 1 = some access; 2 = all access, etc.).

Role Range Specifier

The Role Range Specifier is two numbers separated by a dash (-) to identify the starting role identifier to ending role identifier that is allowed. The user must have a role within this range to pass the check. This specifier only makes sense when role identifiers are set up clustered by security level. For example, if roles 6,7,8,9, and 10 are all supervisory roles, you could use the range 6-10 to include all roles within the range.

 


Example of a Role Range Specifier.

 

  • When creating an ACL, it's a good idea to include the System Administrator role (24) to an ACL. Otherwise, System Administrators won't have access to the component or functionality.
  • The exclamation point is often referred to as a "bang." For example, !0 is read as "bang zero."
  • Spaces are not required within the ACL, but it does break it up for easier reading and allows the text in the field (when in a form) to word wrap. Spaces can also help if you export grid data to a spreadsheet because the data won't be confused as one really long number.
