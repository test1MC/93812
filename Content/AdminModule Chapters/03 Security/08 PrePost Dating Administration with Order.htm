Pre/Post Dating Administration with Order.

Edit Mode: Form

Description: The Pre/Post Dating Administration form and the Pre/ Post Dating Administration with Order PDQ allows Administrators more control over the Pre and Post dating rules in System 7™. This System Administration option is controlled by the Application Setting, Whether to use pre / post dating rules based on order.

The precedence-based system will get the first rule based on the following columns:

  • Role
  • Check Type
  • Program
  • Status From / In
  • Status To
  • Order

For our example, we have, Jim, a Program A Counselor who has role 1001. Keep in mind that Jim's actual roles are 1001 and 0, since every user in the system has role 0.

 

Scenario 1

On April 2nd, if Jim was moving one of his clients into the Program 1 Status 00 as of March 23rd, he would be not be able to. While he's allowed to back date 20 days into the first quarter, the maximum days back is set to 5. March 23rd would be 9 days.

 

Scenario 2

On April 2nd, if Jim was moving a different client into the Program 1 Status 12, as of March 26th, he would be able to. He's allowed to back date 20 days into the first quarter and the maximum days back is set to 5. March 26th would only be 4 days.

 

PDQ Results

Selecting the Pre / Post Dating Administration with Order PDQ option in the Data Tree of the System Administration module loads the related grid in the Electronic Case Folder (ECF).

Note: When you add a new rule, via the Action Row in the grid (the light blue row immediately below the Filter Row), the Pre Post Dating Administration form opens. The values entered in the form appear in the grid. When you select a rule in the ECF this form opens for editing.

 

Columns

Columns marked with an asterisk ( * ) are required for the rule to be valid.

  • Role ID* - This is the role to which the rule applies. When adding a new rule, you must first select the Role ID from the parameter window that appears when you select the Action Row in the grid.
  • Order* - The order in which the rule is applied. The lower the value, the higher precedence of the rule. More generic rules should be at the bottom of the list.

    For example, a rule for role 1001 Program 1status 12-18 should be higher than a rule that is for role 1001 Program 1 and not specific to any status. Please note rules are ordered by lowest to highest (ascending) to show what order the rules are applied.

    It is a good idea when choosing an order to leave space between the rules. That way if you find a need to move a rule or create a new rule between two rules, you can without renumbering existing rules. For the initial rule setup, it would not be bad to have each rule with 100 or more difference in the order. (Rule 1 = order 100, Rule 2 = order 200, etc...)
  • Type* - This is the type of rule. There are three options:
    • All - the rule applies to both status and non status changes.
    • Status Changes Only - the rule applies only to status changes.
    • Non Status Changes Only - the rule applies only to non status changes, in particular case notes.
  • Program Code - This is the program with which the rule is associated. Only one option can be chosen. If you wish to create a generic rule which applies across the system, leave this field blank.
  • Status From / In - For status changes, this is the status the client is going from. For non status changes, this is the client's current status
  • Status To - For status changes, this is the status the client is going to. For non status changes, this is not filled in (It will turn gray).
  • Maximum Days Ahead* - The maximum number of days ahead this rule is allowed to pre-date (forward-date).
  • Maximum Days Back* - The maximum number of days back this rule is allowed to post-date (back-date).
  • # of days in Quarter X Grace Period - These four fields, one each for Quarters 1-4, control the number of days into the quarter that you can back date. For example, if the previous quarter ended on the 30th, and the value for this field is 10, then you can back date to the 20th as long as your date falls within the value entered for the Maximum Days Back.
  • # of days After Quarter X for Backdating - These four fields, one each for Quarters 1-4, control the number of days in the current quarter you are allowed to back date. For example, if the value is 15 days and today is the 14th day into the new quarter, you are allowed to back date per the other established rules. However, if today is the 16th day into the new quarter, you may not back date at all.
  • Note: The system always uses the rules for the previous quarter from the date to check. For example:
    • If the date to check is in the First Calendar Quarter, the rules for Quarter 4 will be used.
    • If the date to check is in the Second Calendar Quarter, the rules for Quarter 1 will be used.
    • If the date to check is in the Third Calendar Quarter, the rules for Quarter 2 will be used.
    • If the date to check is in the Fourth Calendar Quarter, the rules for Quarter 3 will be used
  • Enabled - Whether this rule is active or not.
